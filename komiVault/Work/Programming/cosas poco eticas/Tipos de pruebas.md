Hay tres alcances principales al probar una aplicación o servicio. Su comprensión de su objetivo lo hará determine el nivel de pruebas que realiza en su participación en pruebas de penetración. En esta tarea, cubriremos estos tres alcances diferentes de prueba.

  

**Pruebas de caja negra**![](https://tryhackme-images.s3.amazonaws.com/user-uploads/5de96d9ca744773ea7ef8c00/room-content/6c2f888c43adca666a5850516b67bad6.svg)

Este proceso de prueba es un proceso de alto nivel en el que al evaluador no se le proporciona ninguna información sobre el funcionamiento interno de la aplicación o servicio.

El evaluador actúa como un usuario habitual que prueba la funcionalidad y la interacción de la aplicación o pieza de software. Esta prueba puede implicar interactuar con la interfaz, es decir, los botones, y probar si se desea se devuelve el resultado. No es necesario ningún conocimiento de programación ni comprensión del programa para este tipo de testing.

Las pruebas de caja negra aumentan significativamente la cantidad de tiempo dedicado durante la recopilación y enumeración de información fase para comprender la superficie de ataque del objetivo.

  

![](https://tryhackme-images.s3.amazonaws.com/user-uploads/5de96d9ca744773ea7ef8c00/room-content/1b42f69b00cfb7f24f1e99337c271078.svg)  
**Pruebas de caja gris**

Este proceso de prueba es el más popular para cosas como las pruebas de penetración. Es una combinación de ambos Procesos de prueba de caja negra y caja blanca. El probador tendrá algunos **limitado** conocimiento de lo interno componentes de la aplicación o pieza de software. Aún así, interactuará con la aplicación como si lo fuera eran un escenario de caja negra y luego usaban su conocimiento de la aplicación para tratar de resolver problemas a medida que Encuéntralos.

Con las pruebas de Grey-Box, el conocimiento limitado proporcionado ahorra tiempo y, a menudo, se elige para un ataque extremadamente bien endurecido superficies.

  

  

  

**Pruebas de caja blanca**![](https://tryhackme-images.s3.amazonaws.com/user-uploads/5de96d9ca744773ea7ef8c00/room-content/a7e233d20abe429fb0dda54bb5c6b912.svg)

Este proceso de prueba es un proceso de bajo nivel que generalmente realiza un desarrollador de software que sabe programación y lógica de aplicación. El evaluador probará los componentes internos de la aplicación o pieza de software y, por ejemplo, garantizar que funciones específicas funcionen correctamente y en un plazo de tiempo razonable.

El probador tendrá **completo** conocimiento de la aplicación y su comportamiento esperado y es mucho más tiempo consumir más que las pruebas de caja negra. El conocimiento completo en un escenario de prueba de caja blanca proporciona un enfoque de prueba que garantiza que se pueda validar toda la superficie de ataque.
